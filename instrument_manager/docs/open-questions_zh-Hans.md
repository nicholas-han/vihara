# 待决问题

这些是设计过程中浮现出来的、确实应由创始人拍板的决策。它们都不会阻塞*设计文档的撰写*；标记为 **[gates P0]** 的问题应在我们编写 P0 代码之前回答，因为它们会改动 schema 或示例集合。每条都附上一项建议，以便我们能快速推进。

## Q1 — 计日/年化约定的归属 **[RESOLVED]**

到期时间、`FixedRate`/`FloatingRate` 计日以及 `VarianceLeg.annualization_factor` 的约定应当放在哪里（252 vs 365 vs ACT/365F；加密资产 365 vs 美国 252）？放在 L1 产品上、L0 标的上，还是某个投影配置里？
*建议：* 约定放在 L1 产品上（它属于合约条款），并按资产类别设定合理的默认值。在实现定价投影之前需要确定。

*✅ 已解决（2026-06-15）：放在 **L1 产品** 上，并按资产类别设置默认值（加密资产 365 / 美国 252）。参见 ADR-21。*

## Q2 — 传给 `project()` 的估值日期输入 **[RESOLVED]**

调用方是把单一的 `as_of` 日期传入纯函数 `project()`（最干净——使 `T` 保持为 `asset_pricer` 结构体所需的合约几何输入），还是把"由日期推算 `T`"的逻辑推迟到 `value()` 中与市场数据一并处理？
*建议：* 把 `as_of` 传入 `project()`；保持 `project()` 为纯函数且不含市场数据。在冻结投影接口之前定下来。

*✅ 已解决（2026-06-15）：**把 `as_of` 传入纯函数 `project()`**；`project()` 内部不含市场数据。参见 ADR-22。*

## Q3 — 指数/篮子成分的端点

一个 `Portfolio` 的 `CONSTITUENT_OF` 边只能指向 L0 可观测量，还是也可以指向 L1 挂牌合约（即一种"工具篮子"策略）？
*建议：* 仅限 L0 能保持该层的纯净；把"工具组成的策略"改为建模为一个 L1 `Portfolio` 产品。属于 P1 范畴。

## Q4 — 多腿挂牌 vs 交易场所策略

当某个交易场所把一个结构（例如日历价差）作为单一可交易品挂牌时，它是一个单一的多腿 `Product`，还是两个在 L2 层相连的单腿 `Product`？
*建议：* 以经济性来划线——经济上绑定的结构是一个多腿产品；而把两个各自独立挂牌的产品组合在一起的交易场所订单，则是两个由策略引用相连的 L2 挂牌。属于 P1 范畴（没有任何 P0 交易场所对此有要求）。

## Q5 — 债券和优先股行真的属于 P0 吗？ **[RESOLVED]**

覆盖范围表中列出了附息债券和派息优先股。如果它们属于 P0，那么预留的 `payment_schedules` 载体就必须在 P0 中被*填充*，而不仅仅是预留。
*建议：* 把带现金流排期的产品（债券、优先股股息流）与 OTC 现金流产品一起推迟，除非近期有交易需求；在 P0 中让 `payment_schedules` 保持预留但为空。需要创始人拍板——它直接决定 P0 的 schema 暴露面。

*✅ 已解决（2026-06-15）：**推迟**；`payment_schedules` 在 P0 中保持预留但为空。参见 ADR-23。*

## Q6 — `Rate` / `Volatility` / `Credit` 卫星表是否进入 P0？

这些可观测种类是否要在 P0 中各自获得专用的属性表（利率的计日/期限；波动率的估计量/视野；信用的存活/回收），还是先留在 `assets.metadata` 中，直到被推迟的 swap/varswap/CDS 投影消费它们时再处理？
*建议：* 先放 metadata、被消费时再建卫星表。在为消费方产品定价的那个阶段加入卫星表。

## Q7 — 属性版本化的粒度

是对整个 L2 挂牌/产品行按每次变更整体版本化，还是把高频变动的微观条款（tick/手数/费用）从低频变动的经济条款中拆分出来，纳入各自独立的版本流？
*建议：* 在 P0 的数据量下采用按行版本化（更简单）；如果费用的频繁变动让版本历史变得嘈杂，再重新评估。属于 P1 范畴。

## Q8 — 用于清算的事件总线投递

一旦清算开始消费 `lifecycle_events`，预留的 `sequence_no` 加上"经由幂等重放的事务性发件箱"是否足够，还是需要一套 CDC / LISTEN-NOTIFY 机制？
*建议：* 这些列现在已预留；在构建清算模块时再敲定投递契约。推迟处理。
