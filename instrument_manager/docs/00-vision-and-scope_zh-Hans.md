# 愿景与范围

## 使命

`instrument_manager` 是一个**万物交易所 / 万物经纪商**的静态数据 / 参考数据核心。它的职责是用一套统一连贯的模型，管理*每一种*可交易的金融产品——证券与衍生品皆然——以及*每一个*可定价但不可交易的可观测量（指数、利率、事件、波动率）。

从用户的视角（无论是内部或外部交易员、投资者，还是调用我们 API 的系统）来看，这个模块正是让平台其余部分能够谈论某个金融工具的根基：它是什么、它暴露于何种风险、它在哪里交易、如何命名、以及如何定价。它是市场数据、交易、订单、持仓，乃至最终的清算与结算之下的那一层。

## 两大抱负，同一根基

平台有两个终极抱负，而**两者随着时间推移都属于本模块的范围**：

1. **衍生品定价与风险。** 一套足够丰富的产品定义，能够馈入 [`asset_pricer`](../../asset_pricer) 的估值引擎，并支持风险聚合。这一点在 P0 中即被落实。
2. **一个具备清算与结算的全生命周期交易所。** 在金融工具与挂牌模型之上构建持仓、成交、保证金、清算与结算。

抱负 2 的交易后（post-trade）机制被有意**推迟**——它可以稍后再建。P0 所承诺的是*为它留出干净的空间*：预留的接缝、不透明 id（opaque-id）边界，以及一条事件主干，让被推迟的清算/结算层能够接入其中，而无需强制迁移 L0–L2 核心。关于究竟哪些是预留、哪些是已构建的，请参见 [`60-lifecycle_zh-Hans.md`](60-lifecycle_zh-Hans.md)。

## 一段话讲清模型

一个金融工具不是单一的事物；它是一个**由四层构成的栈**外加两个横切关注点。**L0** 参考数据 / 可观测量是底层标的（资产、指数、利率、事件、波动率）。**L1** 产品/合约是与场所无关、与对手方无关的经济定义——以**强类型的支付腿（payout legs）组合**来承载（受 CDM 启发但更为精简），这正是馈入定价的部分。**L2** 挂牌/可交易是某产品在特定场所挂牌后的形态，包含全部微观结构（代码、最小变动价位、手数、费用、日历、状态）。**L3** 分类是*从经济性派生*而来，从不手工编写。横贯所有层之上的是**身份与符号体系**（不透明 id + 带生效日期的标识符映射）与**生命周期与生效日期**（静态数据其实是缓慢变化的数据）。完整全貌见 [`10-layered-model_zh-Hans.md`](10-layered-model_zh-Hans.md)。

## 范围——P0 涵盖什么

P0 必须在不做特殊处理的前提下，表达我们今天实际能够获取来源的、可定价或可交易的全部宇宙：

- **现货 / 持仓：** 股票（普通股、优先股）、加密币种与代币、FX、包装型/代表型代币（Ondo RWA、Hyperliquid Unit）
- **可观测量：** 指数、利率、事件、波动率（作为 L0，而非产品）
- **期货：** 有到期日的与永续的；实物交割与现金结算；**线性与反向**（币本位保证金）——反向永续被视为一等的旗舰案例
- **期权：** 欧式 / 美式 / 百慕大式；香草 / 二元 / 障碍；现金与实物结算；通过产品嵌套实现的**期货期权（options-on-futures）**
- **结构化 / 池化：** ETF 与基金（对某个池/NAV 的索取权）、分类型预测市场（结果分割）、方差互换
- **ETF / 投资基金**以及金库/基金份额

具体的基准是 v1 示例宇宙，加上覆盖表标记为所需的那些行（反向永续、分类型市场、方差互换、债券、优先股、FX、资金费率/SOFR/VIX 可观测量）。每一个产品 → 其 L1 组合 → 其定价器映射，都在 [`20-product-economics_zh-Hans.md`](20-product-economics_zh-Hans.md) 的覆盖表中逐一列举。

## 范围——被推迟，但模型必须能弯曲

- **场外互换（IRS、CDS、TRS、swaption）。** 不紧迫，且目前尚无来源——但 L1 的承载体是一个多腿支付组合，正是为了让这些产品*稍后通过组合即可纳入，零结构性重塑*（腿目录中已包含它们所需的强类型腿；只有定价引擎与支付计划承载体被推迟）。参见 [`20-product-economics_zh-Hans.md`](20-product-economics_zh-Hans.md)。
- **持仓、成交、清算、结算、保证金。** 已预留，未构建。参见 [`60-lifecycle_zh-Hans.md`](60-lifecycle_zh-Hans.md)。

## 非目标（暂时）

- 我们**不**采用完整的 ISDA/CDM Rosetta DSL、法律协议模型或交易后事件机制。我们借用 CDM 的*产品分解*理念（可组合的强类型支付、标的嵌套、身份/经济性分离、推断式分类），到此为止。
- 我们在 P0 中**不**构建场外互换定价引擎、曲线/违约率模型或波动率互换引擎——L1 承载体能够表达这些产品，但在引擎存在之前，它们会投影为 `NonPriced` / `Unsupported`。
- 我们在 P0 中**不**构建清算/结算/持仓层。

## 与仓库其余部分的关系

- [`asset_pricer`](../../asset_pricer) 拥有估值。`instrument_manager` 产出类型良好的经济条款，并将其*投影*到 `asset_pricer` 的合约结构体中；它从不定价。该边界在 [`80-pricing-integration_zh-Hans.md`](80-pricing-integration_zh-Hans.md) 中规定。
- PostgreSQL 是缓慢变化的参考数据的记录系统（system of record）；C++ 核心是内存模型、校验的单一事实来源（Single-Source-of-Truth，通过 pybind11 共享给 Python），以及所有语义的归属地（分类、投影、图遍历）。该边界见 [`70-persistence-and-cpp_zh-Hans.md`](70-persistence-and-cpp_zh-Hans.md)。

## 与 v1 的关系

v2 是一次全新的构建，它承袭了 v1 的良好骨架——不透明的稳定 id、Postgres 作为 SoT + 内存注册表、通过 pybind11 共享的 C++ 校验 SoT、派生的关系图、场所分段，以及"类型是涌现的，而非枚举的"原则——并将它们重组进上述分层模型。v1 分支（`instrument-manager-v1`）已归档；v2 在就绪时合并进 `main`。相较 v1 最大的单一变化：L1 承载体从单一的 `payoff_form` 枚举 + JSONB，转为一个**强类型的 13 成员支付腿组合**，并且产品经济性（L1）与场所挂牌（L2）相互拆分。

已确认与已提议的决策记录在 [`decisions_zh-Hans.md`](decisions_zh-Hans.md) 中。
